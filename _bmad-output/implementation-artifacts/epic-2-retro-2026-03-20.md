## Successes

- **"Something working" mandate executed.** Story 2.1's primary success criterion was explicitly stated as: "a human can run `cargo run`, see a full-screen TUI, and press `q` to quit cleanly." Delivered. This was the north star lesson from Epic 1's retro, applied immediately.

- **Code review caught a critical CPU bug.** `Duration::ZERO` in the event poll loop would have caused 100% CPU usage at idle. Caught and fixed in Story 2.1's review before merge. The review process paid for itself on the first story.

- **Scope discipline held.** Stories 2.2, 2.3, and 2.5 each touched exactly one or two files. Clean, reviewable, no scope creep.

- **Proactive safety-net test added.** Story 2.2's developer added `test_float_value_displays_as_decimal` beyond the spec — a guard for the documented FBig gotcha from Epic 1. Unplanned, but exactly the right instinct.

- **Story 2.4 delivered clean despite complexity.** The largest story in the epic (5+ files, full keybinding wiring, alpha mode, 35+ tests) landed with zero code review findings.

- **Epic 2 is where rpncalc became a calculator.** First time a human could pick it up and perform a calculation. The engine was the foundation; Epic 2 was the delivery.

## Challenges

- **Spec contained a performance error.** The Dev Notes in Story 2.1 specified `event::poll(Duration::ZERO)` — which was faithfully implemented and then caught in code review. A wrong spec is more expensive than a wrong implementation: the developer has no reason to question it. Two of the four code review findings were spec-level errors, not dev errors.

- **Terminal error-path not covered in spec.** `setup_terminal` had a missing error path: if `execute!(EnterAlternateScreen, Hide)` failed after `enable_raw_mode()`, the terminal was left in raw mode. This is a subtle, hard-to-reproduce class of bug. Not in the spec; caught in review.

- **Integration test placeholders never replaced.** Committed in Epic 1's retro. Carried through all four epics. Real integration tests for session roundtrip and keybinding sequences would have provided seam-level coverage that unit tests cannot.

---

## Key Lessons Learned

1. **Spec errors are more expensive than code errors.** When Dev Notes specify a wrong pattern, the dev implements it faithfully, review catches it, and a fix cycle follows — at significant cost. For performance-sensitive patterns (event loops, timers, buffering), cross-reference the architecture doc before writing Dev Notes.

2. **Integration test debt compounds silently.** 73 new unit tests over Epic 2, but no new integration test coverage. Regressions at the seams (key → action → engine → render) would only be caught manually. Commit to at least one real integration test per epic.

3. **Known footguns deserve "guard test prompts" in story specs.** The FBig float display bug is documented but relies on developer memory to test. Story Dev Notes should have a "Known Footguns" subsection that explicitly prompts the developer to add guard tests for documented gotchas.
